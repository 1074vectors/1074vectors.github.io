3D programming always had an artistic part and a precise engineering part.

Let us talk about engineering first. It's been challenging to retain quality of 3D without losing accuracy and performance. How to reduce repeated computing to improve performance? Simple! Store some results. It’s a balance between Memory usage vs Computing for a better performance. This old theory is grabbing new attention nowadays. With the distributed nature and low-capacity of the new generation head-end devices, new parameters are evolving and balancing different aspects has become more challenging. As batteries are involved, the metrics like quality/watt and quality/degree-Celsius too need to be considered. As the FLOPS/$ for computing can be split between local and cloud enabled internet/intranet resources, the load division between them too is another point of consideration. In a nutshell, a good engineering solution cannot be achieved by tuning a single aspect alone.

We do programming. Specializes in “behind the screen technologies” of interactive graphics. More than coding, our job includes technology selection, architecting and technical support after deployment. Normally the technical support is provided against queries specific to certain hardware platforms (eg:- GPU).

Global Software development is not that easy. Especially when it comes to 3D. Our ‘e’ medias still has a lot to grow in communicating subjects around 3D. In this scenario, it’s not that easy to start an engagement that will result in a win-win situation.

We value our time and that of our clients too. We value the money budgeted for the project. Hence we do ‘outcome’ based model of business (a.k.a turnkey contracts). There is no man power discussions as we value the result than charging for man-hours. However still… to start any development it is important to know the volume initially itself.

We adopt two methods to START the engagement. One is unpaid, while the other is marginally charged.

1. Proposal (unpaid)

2. Technical analysis report & estimate

Proposals: Proposals are executed for a first level review of the requirements in a technology context. Obviously it will propose a solution in a document. It will have technology details and rough understanding of the requirements. It may also include a ball park estimate. It depends on the clarity of the solution, the complexity of targeted hardware and known varieties of input data. As our experts are normally engaged, scheduling for a proposal may take time. So this unpaid method is apt for those who have flexibility in time-to-market.

Technical Analysis report with estimation: A document is delivered which contains a proposed solution with refined requirements, Technology selection, project schedule and a price estimate. Project will have an FS stage (functional specification stage) after this. This stage gives clarity on the features of the solution. But Technical analysis report is made after testing the potentially risky areas. Though the features are not fully formed, normally we can expect a good success rate, which is around 86% for us. The rest had requirement revamping during the course of the time. Depending on the complexity, the price ranges from 3000 USD to 30,000 USD.

Which model is apt? For an industrialist, it is better to start with the latter engagement model as the estimates here are accurate than a mere ballpark. This will help in proper budgeting which may again need to plan cash-flow. Though this is a paid method, the value of the report is not fully charged as we are also considering things in a start-up mind-set. After all, the success of this report is the beginning of a wonderful industrial product. If you are just seeking for a possibility of a solution for an initial planning as an academic exercise, first method is sufficient though the schedule is bit unpredictable.
